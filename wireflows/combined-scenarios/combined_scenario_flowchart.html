Phase 1 — Discovery: How does an analyst know they need to combine?
Entry Point A
A
Planned national negotiation
Analyst is assigned a multi-market contract (e.g. Tenant). Knows from the start they'll need regional rollups. Models by contract office intentionally.
Workflow: Model → Combine → Roll up by region → Repeat
Entry Point B
B
Post-hoc rollup need
Analyst modeled ASCs and hospitals separately. Stakeholder asks for a combined view. Analyst realizes existing individual scenarios can be merged.
Workflow: Existing scenarios → Discover combine option → Merge
Entry Point C New
C
Bulk selection from search page
Analyst filters the scenario list by system/region, selects multiple rows, hits "Combine Selected." Discovery and assembly happen in one action.
Workflow: Filter → Select → Combine (no ID hunting)
⚠ Business question: Should analysts be able to create a "negotiation group" or "project" to pre-organize scenarios before combining? This would solve Entry Points A and B upstream — scenarios know they belong together before the combine step.

This view maps the business rules, constraints, and data relationships that govern the combined scenario feature. Use this to align engineering scope and surface edge cases before design handoff.
Compatibility Rules (Hard Gates)
Effective Date Must match
Proposed Effective Date Must match
Claim Date Range Must match
Claim Version Must match
Lines of Business Must match (debated)
Future Value Setting Must match
Rationale: Rules exist because you cannot build a single scenario from scratch with mismatched dates/versions. The combine must be mathematically equivalent to building from scratch.
Data Model — What Gets Stored
Today
Combined scenario stored as a regular scenario. No link back to source IDs. Name convention is the only signal.
Proposed — Provenance record
Store: source_scenario_ids[ ], combined_on_date, negotiation_group, is_combined: true. Displayed in report. Enables in-place management.
Open question — Staleness
If source scenario A is modified after combined scenario C was built from [A, B] — should C be flagged as stale? Does the system need to track source version at combine time?
Nested combine
Can a combined scenario be a source in another combine? Currently yes (combine C then use C's ID in a new combine). Should the system resolve the full source tree or treat combined IDs as opaque?
User Types & Access Patterns
SA
Senior Analyst (Skander)
Power user. Manages national negotiations. Combines 5–10 scenarios at a time. Maintains Excel tracker today. Needs provenance + in-place management most.
MA
Mid-level Analyst
Combines ASCs + hospitals for a single market. Lower frequency. Most at risk of being blocked by compatibility errors they don't understand. Needs pre-flight clarity most.
M
Manager / Stakeholder
Reads combined reports. Wants to know: which facilities, which markets, what trend. Doesn't combine — consumes outputs. Benefits from provenance card and region summary.

In scope (this cycle)

These are the unresolved business questions that should be answered before finalizing the design or handing off to engineering. Each one has a recommended owner and decision approach.
1
Should we introduce a "Negotiation Group" or "Project" concept upstream?
Strategic
Context
Skander uses Excel to track which scenarios belong to which negotiation. The tool has no concept of grouping. Without it, analysts will always need an external tracker even after the combine enhancements ship.
Options
A. Add a "Negotiation Group" tag to scenarios (lightweight, low effort)
B. Build a project/folder hierarchy — scenarios belong to a project (larger scope)
C. Don't build it — multi-select on search page is enough for now
Recommendation
Start with Option A — a simple tag/label field on scenario creation. Low effort, eliminates the Excel tracker, doesn't require a new data model. Can evolve into Option B later.
2
What happens when a source scenario is modified after a combine was run?
Data integrity
Context
Combined scenario C was built from [A, B]. An analyst later modifies scenario A's timeline. C is now stale — it reflects the old version of A. Currently there is no indication of this.
Options
A. Show a "stale" warning on the combined scenario report when any source was modified after last run
B. Store source scenario version/hash at combine time, detect drift automatically
C. No system tracking — analyst is responsible for re-running when needed
Recommendation
Option A as MVP — compare source scenario "date modified" to the combined scenario's last run date. Show a simple "Source updated — consider re-running" banner. No version hashing needed initially.
3
Should LOB cross-combine (OAP + New Business) be enabled?
Edge case
Context
A facility models OAP and New Business in separate scenarios with different LOBs. Today you cannot combine them. Skander said this is not a priority — but acknowledged it as a real limitation.
Risk
Enabling this breaks the mathematical equivalence guarantee — the combined output would no longer be identical to building from scratch with both LOBs, because the base assumptions differ.
Recommendation
Keep blocked. Document as a known limitation in the UI with a clear explanation. Investigate usage data — if more than 5% of failed combines are LOB conflicts, revisit.
4
How should nested combines be handled — flat or tree-aware?
Architecture
Context
Skander combines [A, B] → C, then combines [C, D] → E. Today the system treats C as an opaque scenario ID. The provenance of E won't show A and B — only C and D.
Options
A. Flat — provenance shows immediate sources only (C, D). User must navigate to C to see A, B.
B. Tree — provenance expands fully: E → [C → [A, B], D]. Shows full lineage.
Recommendation
Start with Option A (flat) with clickable links. Analyst can navigate the chain. Full tree display is a nice-to-have — adds complexity without clear demand. Revisit if analysts complain about lineage depth.
5
Who else needs to be interviewed before design handoff?
Research
Context
Skander is a technically advanced power user. He has normalized workarounds. We do not know if his workflow represents the median analyst using or trying to use this feature.
Gaps to fill
• Mid-level analyst who tried and failed to use combine
• Manager who reads combined reports (not the builder)
• Analyst at a different contract office — is the Excel tracker universal?
Recommendation
1–2 more interviews before finalizing the search page redesign. The combine flow enhancements can proceed — those are clearly validated. The search page changes need broader signal.
