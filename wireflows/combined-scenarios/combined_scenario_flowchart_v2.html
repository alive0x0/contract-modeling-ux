Business Logic & Constraints

Compatibility rules, data model evolution, user types, and scope boundaries

🔒 Compatibility Rules — What must match for scenarios to combine

Rule Status Why it matters Notes
Effective Date Must Match Contract periods must align for the combined model to make actuarial sense Exact match required
Proposed Effective Date Must Match Rate proposals apply to the same forward period Exact match required
Claim Date Range Must Match Experience data windows must be consistent for valid comparison Start + end date both matched
Claim Version Must Match Mixing paid vs incurred or different run-out periods distorts combined results e.g. Paid, Incurred
Lines of Business Debated OAP vs New Business have different pricing structures — combined view may be misleading Currently blocked; under review
Future Value Setting Debated Scenarios projecting different future periods may not be meaningfully combined Edge case; escalate to pricing team

Open Questions

5 unresolved decisions requiring stakeholder input — with options and current recommendations

Question 01
Negotiation Group Tag
Strategic
A Add a Negotiation Group field to scenarios at creation time — pre-organizes everything before combining.
B Apply a tag/label to existing scenarios retroactively — lighter touch, flexible, but less structured.
C No pre-organization — rely solely on search filters and multi-select at combine time.
💡 Recommendation
Start with Option B — a tag label. Lower engineering cost, validates the need before committing to a structured field. Can evolve to Option A if adoption is high.
Question 02
Source Modified After Combine
Data Integrity
A Show a date-modified comparison banner on the combined scenario — passive warning, analyst decides what to do.
B Block re-export / sharing until analyst acknowledges staleness — more aggressive guardrail.
C Ignore it — document as known limitation, out of scope for v1.
💡 Recommendation
Start with Option A — a non-blocking banner comparing source modification timestamps to the combined run timestamp. Non-destructive, surfaces signal without blocking workflow.
Question 03
LOB Cross-Combine
Edge Case
A Keep blocked — OAP + New Business cannot be combined. Hard rule enforced by compatibility check.
B Allow with warning — show prominent disclaimer that cross-LOB combined results require careful interpretation.
C Allow with separate outputs — run combined model but surface LOB breakdown separately in results.
💡 Recommendation
Keep blocked (Option A) for v1. Document the rule clearly in the compatibility error message. Revisit after interviewing mid-level analysts and pricing actuaries.
Question 04
Nested Combines — Flat vs Tree
Architecture
A Flat list — always expand nested combines to their leaf-level source IDs. Provenance card shows all originals, no nesting.
B Tree view — show the actual hierarchy of combines. Combined-of-combined shows as a tree node with children.
C Block nested combines — disallow combining a combined scenario entirely.
💡 Recommendation
Start with Option A — flat list with clickable source links. Simpler to implement, covers the primary use case. Tree view is a future enhancement if analysts need hierarchy visibility.
